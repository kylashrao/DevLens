What a JWT actually is

A JSON Web Token has three base64url-encoded segments separated by periods: a header (describing the signing algorithm), a payload (the actual claims — user ID, roles, expiration, etc.), and a signature. Anyone can decode the header and payload without any secret, since base64 is an encoding, not encryption — which is exactly what this tool does. Only the signature requires a key, and only a server with that key can confirm the token hasn't been tampered with.

Frequently asked questions

Does this tool verify the JWT signature?

No. Verifying a signature requires the secret or public key used to sign the token, which this tool never asks for or needs. It only decodes and displays the header and payload so you can inspect the claims.

Is it safe to paste a production JWT into this tool?

Decoding happens entirely in your browser using JavaScript, and the token is never transmitted anywhere. That said, JWT payloads are only base64-encoded, not encrypted, so treat any token holding sensitive data with the same care you would the data itself — the same caution applies to any JWT decoder, including this one.

What does "expired" mean here?

If the payload includes a standard "exp" claim (expiration time, in seconds since the Unix epoch), DevLens compares it to your device's current clock and flags the token as expired if that time has passed. If your system clock is wrong, this comparison will be too.
